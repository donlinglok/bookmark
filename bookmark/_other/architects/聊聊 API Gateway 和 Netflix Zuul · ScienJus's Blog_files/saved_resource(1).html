            感谢你的建议，关于这点，详细来看我觉得可能会细分成两个部分：

            1. 我的服务由哪些 API 组成：这种情况通过注解与代码强绑定的好处就是不易犯错，作为基础版本能保证与实际提供服务的 API 完全相同。
            2. 我该怎么管理这些 API：临时重定向或者关闭，肯定是走配置更加方便，上文中的转发也是通过读取动态数据源进行配置的。

            > 你可以在每个服务启动的时候把自己的API 注册到某个配置服务上，然后要修改的话，可以去直接改配置会比较灵活。

            在我目前的想法中，基于上面的细分，可能还是会做一个两层的覆盖结构，让人工创建的配置可以任意覆盖掉系统生成的配置，但不是修改，既能保证临时方案的生效速度，也需要尽量保证需要长期维持的方案尽量同步到代码中，减少维护的成本。

                                                  ScienJus Mod 梁鹏 • 5 years ago

                                                  这个问题好！

                                                  在使用 Gateway 之前我们已经有了跨语言的各种微服务，以及通用的 LB 设施，这个时候由于无法使用 Erueka，导致使用 Zuul 的性价比很低。

                                                  但是我们依旧想在 Gateway 这一层做更多事情：多种鉴权适配、Rate Limiting、Rewrite、反作弊、统计、监控等等，如果在原先的 LB 上进行业务开发一个是用 Lua 对当前技术栈来说效率很低，另一个是一个通用的 LB 本不应该有这些业务逻辑，所以就拆出来用 Zuul 实现了。

                                                  所以在这点上，详细来说我们用的是 Netflix Zuul，并且只用了其 Filter 模型，而不是 Spring Cloud Netflix Zuul，一个和 Spring Cloud 完全兼容的 Gateway 实现。
